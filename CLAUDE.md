## Repository layout

Codedrift is a code-intelligence / diff-review app. There is **no root `package.json`** — the two apps are developed and run independently from their own directories.

- `codedrift-backend/` — Node + Fastify + TypeScript API. Reads target git repos via `simple-git`, computes per-changeset diffs (`GET /review/:reviewId/changeset/:changesetId/diff`), manages registered repositories (add / list / delete + list branches) and multi-repo reviews, and persists both in a local SQLite database (`better-sqlite3`).
- `codedrift-frontend/` — React 19 + Vite + TanStack Router/Query + Tailwind v4 + Shadcn UI. Renders the diff review UI.
- `api-collection/Codedrift/` — Bruno (OpenCollection) API collection; keep in sync with backend endpoints.

## Commands

### Backend (`cd codedrift-backend`, Node >= 22)
- `pnpm dev` — `tsx --watch src/main.ts`, server on port **3000**
- `pnpm build` — `tsc`
- `pnpm typecheck` — `tsc --noEmit`
- `pnpm start` — `node dist/main.js`
- No test runner is configured. Lint via `eslint .`; Prettier runs on pre-commit (husky + lint-staged).

### Frontend (`cd codedrift-frontend`)
- `pnpm dev` — Vite dev server
- `pnpm build` — `tsc -b && vite build`
- `pnpm lint` — `eslint .`
- `pnpm preview` — preview production build
- No test runner. Prettier runs on pre-commit. After changes, the frontend AGENTS.md asks for `tsc` + prettier; do not run vite/the browser to check visuals unless asked.

## Architecture & data flow

The backend operates on **live git repositories on disk**: diff/git operations are stateless and re-run per request. Registered repositories and reviews are persisted in a local **SQLite** database (`better-sqlite3`) at `~/.config/codedrift/codedrift.db`, opened at startup by `src/db/database.ts` — which creates the config dir, sets the `journal_mode=WAL` / `foreign_keys=ON` / `busy_timeout=5000` pragmas, and creates the schema inline (`repositories`, `reviews` — with a nullable markdown `overview` column, the `review_repositories` join table, `changesets`, `changeset_files` — each file linked to a `repository_id` and carrying a `summary`, and `changeset_file_comments` — line-anchored review comments with a `side` of `old`/`new`). All `id`s are UUIDv7 `TEXT PRIMARY KEY`s generated app-side via the `uuidv7` package (not autoincrement). Data access goes through small stores in `src/db/` (`repository-store.ts`, `review-store.ts`, `changeset-store.ts`).

1. `src/services/diff-service.ts` is parameterized by `(repoPath, baseRef, headRef)`: it shells git through `simple-git`, parses raw diff output, normalizes paths, classifies each file (added / deleted / moved / changed), and fetches full file content at base and head. Shared shapes live in `src/@types/diff.ts` (`DiffFileData`, `DiffChangeType`). `src/services/changeset-diff-service.ts` builds on it: for a `(reviewId, changesetId)` pair it diffs each involved repository at the review's refs, filters to the changeset's files, and merges in each file's `summary`/`comments`/repository identity (`ChangesetDiff` in `src/@types/changeset-diff.ts`).
2. `src/main.ts` is the Fastify entry point (permissive CORS via an `onRequest` hook allowing `GET, POST, DELETE, OPTIONS`). At startup it opens the SQLite DB (`import "./db/database.ts"`) and registers per-resource route plugins via `fastify.register`.
3. **Routes** live in `src/routes/` as Fastify plugins (`FastifyPluginAsync`) — e.g. `repository-routes.ts` exports `repositoryRoutes`, registered in `main.ts`. `POST /addRepository` validates that the body `repositoryPath` is an absolute git repo (`src/services/git-service.ts`, via `simple-git` `checkIsRepo`) and persists `(repositoryName, path)` to the `repositories` table through `src/db/repository-store.ts` (the `path` column is `UNIQUE`, so duplicates return 409). `repository-routes.ts` also serves `GET /repositories` (list all), `GET /repository/:repositoryId/branches` (local branch names via `RepositoryService.getBranches` → `simple-git` `branchLocal()`), and `DELETE /repository/:repositoryId` (404 on unknown id). Store helpers: `saveRepository`, `getRepositoryById`, `listRepositories`, `deleteRepository`. `review-routes.ts` (`reviewRoutes`) covers reviews: a review has a **name** and **one-or-more repositories, each with its own base/head ref** (no longer one-repo-per-review), persisted across the `reviews` + `review_repositories` tables via `src/db/review-store.ts` (`saveReview` runs inside a `better-sqlite3` transaction; plus `getReviewById`, `getReviewInfo`, `setReviewOverview`, `getReviewOverview`, `listReviews`, `deleteReview`). Endpoints: `POST /addReview` (body `{ name, repositories: [{ repositoryId, baseRef, headRef }] }`; validates name, non-empty list, distinct base/head per repo, existing repos, no dup repos), `GET /reviews` (all reviews with nested repositories), `GET /review/:reviewId/changesets` (a review's changesets with their files — each file has `repositoryId`/`repositoryName`/`filePath`/`summary`/`comments` — via `changeset-store.getChangesets`; always returns an array, never 404), `GET /review/:reviewId/overview` (the review's markdown overview, `null` until set; 404 on unknown review), `GET /review/:reviewId/changeset/:changesetId/diff` (the changeset's `ChangesetDiff` via `ChangesetDiffService`; 404 on unknown review/changeset), `DELETE /review/:reviewId` (404 on unknown id).
4. All endpoints respond with the `ApiResponse<T>` wrapper `{ success, message, data }` from `src/@types/api-response.ts`.
5. `src/tools/changeset-tools.ts` exposes review-scoped operations as plain functions (no `ai`-SDK): `getReviewInfo` (a review's repositories with their id + path + base/head ref), `addReviewOverview` (set/overwrite the review's markdown overview), `addChangeset` (persist a named changeset for a review; takes **absolute** file paths, resolves each to a review repository by longest path-prefix match and stores the repo-relative path + `repository_id`, plus a per-file `summary` and optional line-anchored `comments` `{ lineNumber, side: "old"|"new" (default "new"), comment }`; rejects paths outside the review's repos and files already in another changeset of that review, keyed per `(repository, path)`), and `getChangesetFiles` (absolute paths of all changeset files of a review). Changesets persist in the `changesets` + `changeset_files` + `changeset_file_comments` tables via `src/db/changeset-store.ts` (`saveChangeset`, `getChangesetFileKeys`, `getChangesets`, `getChangesetById`; reads are exposed over REST, writes stay MCP-only). These functions are exposed as **MCP tools** (not REST) by an MCP server at `POST /changeset-mcp` — a stateless Streamable HTTP transport (`@modelcontextprotocol/sdk`) mounted as the `mcpRoutes` plugin (`src/routes/mcp-routes.ts`, registered in `main.ts`); the server/tool wiring lives in `src/mcp/changeset-mcp-server.ts` (tools `get_review_info`, `add_review_overview`, `add_changeset`, `get_changeset_files`, each keyed by `reviewId`; `ChangesetInputError` → tool error). Register a client with `claude mcp add --transport http codedrift-changeset http://localhost:3000/changeset-mcp`.

On the frontend:
1. TanStack Router uses **file-based routing** — routes live in `src/routes/`, and `src/routeTree.gen.ts` is **auto-generated** by `@tanstack/router-plugin` (do not edit by hand; it regenerates on `vite dev`/`vite build`). Main routes: `/` redirects to `/dashboard/repositories`; a `/dashboard` layout (`src/routes/dashboard/route.tsx`) renders a fixed nav sidebar (`src/components/dashboard-sidebar.tsx`, links to repositories/reviews) alongside its child routes `/dashboard/repositories` (repository CRUD), `/dashboard/reviews` (review CRUD), `/dashboard/reviews/$reviewId` (per-review detail), and `/dashboard/reviews/$reviewId/changesets/$changesetId` (the per-changeset diff viewer — a `route.tsx` layout + `index.tsx` pane).
2. The review detail route (`src/routes/dashboard/reviews/$reviewId/index.tsx`) fetches the review's changesets (`["changesets", reviewId]`). With no changesets it renders `src/components/review/review-setup.tsx` — the copy-pasteable AI-agent instruction (create the review overview first via `add_review_overview`, then changesets with absolute paths, per-file summaries, and line comments). With changesets it renders the review **overview** (`["reviewOverview", reviewId]` → `src/components/review/review-overview.tsx`, markdown via the shared `src/components/markdown-content.tsx`, where ` ```mermaid ` fences render through `src/components/review/mermaid-diagram.tsx` — mermaid v11 + `react-zoom-pan-pinch` for zoom/pan) with the clickable changeset list beside it (`src/components/review/changeset-list.tsx`).
3. The changeset diff layout (`.../changesets/$changesetId/route.tsx`) fetches `GET /review/:reviewId/changeset/:changesetId/diff` (keyed `["changesetDiff", reviewId, changesetId]`) and pushes `data.files` into the Zustand store `src/store/diff-view-store.ts` (`diffFiles: ChangesetDiffFile[]`, `selectedFileId`). The store auto-selects the first file and keeps the selection valid across navigation.
4. The diff file tree (`src/components/file-browser.tsx`) renders changed files grouped by directory (paths are prefixed with the repository name) with color-coded change types; clicking a file sets `selectedFileId`, and the pane (`.../changesets/$changesetId/index.tsx`) renders it with `@git-diff-view/react` (split view) — the file's `summary` shows above the diff, and its line-anchored `comments` render inside the diff via `extendData`/`renderExtendLine` (keyed by 1-based old/new-file line numbers).
5. The `/dashboard/repositories` and `/dashboard/reviews` route files own their CRUD (TanStack Query + `useMutation`, raw `fetch` via `src/service/{repository,review,changeset}-service.ts`, base URL `VITE_API_BASE_URL`, default `http://localhost:3000`) and the add/delete shadcn `Dialog`s, delegating list rendering to presentational components (`src/components/repository-list.tsx`, `src/components/review/review-list.tsx`). Creating a review names it and adds one-or-more repositories — each row (`src/components/review/review-repository-entry.tsx`) loads that repo's branches (`GET /repository/:id/branches`, keyed `["branches", repositoryId]`) and picks base/head; the dialog's per-row form state uses the `ReviewEntry` type (`src/@types/review-entry.ts`). Success toasts fire via **Sonner** — `<Toaster/>` is mounted once in `__root.tsx`. Frontend shared types mirror the backend in `src/@types/` (`repository.ts`, `review.ts`, `changeset.ts`, `changeset-diff.ts`, `api-response.ts`).

## Conventions not in AGENTS.md

- Frontend path alias `@/*` → `./src/*` is set in both `tsconfig` and `vite.config.ts`. The backend has no `paths` alias — `@types` there is a real directory (`src/@types/`) imported by relative path.
- Backend uses native ESM with `.ts` import extensions (`allowImportingTsExtensions`, `rewriteRelativeImportExtensions`); keep the `.ts` extension in relative imports.
- Shared/reused types go in the `@types` folder in each app.
- Add shadcn UI components by running the CLI from `codedrift-frontend/` (`pnpm exec shadcn add <component>`) — don't hand-write files into `src/components/ui/`. A generated component may import deps that don't fit this light-only Vite app (e.g. `sonner.tsx` pulled in `next-themes`); strip those imports and remove the dep.
- The changeset tools are exposed via MCP using `@modelcontextprotocol/sdk` (stable v1.x; stateless Streamable HTTP transport at `POST /changeset-mcp`). When building or debugging LLM/tool features here, prefer the latest Claude models.
